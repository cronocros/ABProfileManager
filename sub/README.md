# Sub Agent Team

이 폴더는 `ABProfileManager` 작업을 4명 체계로 굴리기 위한 역할 정의와 운영 규칙을 담습니다.

## 팀 구성

- `control-lead.md`
  - 요청 분류, 우선순위 결정, 세 실무 에이전트 결과 승인/반려
- `doc-maintainer.md`
  - 문서 업데이트와 문서 검수 전담
- `source-implementer.md`
  - 실제 소스 수정, 리팩터링, 버그 수정, 기능 구현 전담
- `source-reviewer.md`
  - 소스 변경 검수와 회귀 위험 점검 전담

## 기본 원칙

- 문서 변경은 `doc-maintainer`가 먼저 책임지고, 소스 사실관계가 걸리면 `source-reviewer` 확인을 받습니다.
- 소스 변경은 `source-implementer`가 실제 수정을 맡고, `source-reviewer`가 승인 전 검수를 맡습니다.
- 사용자 문서나 릴리스 노트 영향이 있으면 `doc-maintainer`가 후속 반영을 맡습니다.
- 최종 승인과 작업 종료 판단은 항상 `control-lead`가 합니다.
- 메인 UI 레이아웃, profession overlay, 지도 overlay는 회귀 민감 영역으로 간주합니다.
- 근거 없는 추측 문서화는 금지합니다. 문서는 반드시 현재 소스와 릴리스 상태를 기준으로 맞춥니다.

## 표준 흐름

1. `control-lead`가 요청을 `문서`, `소스`, `혼합`으로 분류합니다.
2. 주 담당 에이전트를 지정하고 산출물 형식을 고정합니다.
3. 소스 작업이면 `source-implementer`가 수정하고 `source-reviewer`가 교차 검수합니다.
4. 충돌이 있으면 `control-lead`가 우선순위를 정합니다.
5. 최종 보고는 `control-lead`가 변경 요약, 위험, 남은 확인 항목까지 묶어서 냅니다.

## 산출물 규칙

- 문서 담당은 변경 파일, 불일치 수정 근거, 남은 문서 TODO를 남깁니다.
- 소스 수정 담당은 변경 파일, 수정 의도, 영향 범위, 필요한 후속 문서 반영을 남깁니다.
- 소스 담당은 위험도 순 검수 결과, 회귀 포인트, 필요한 검증 명령을 남깁니다.
- 총괄은 승인 여부, 반려 사유, 다음 액션을 한 문단으로 정리합니다.
